# 专题：工程验证与发布面

相关节点：[[01-项目总览]]、[[01-仓库地图]]、[[模块-公共API与Prepared模型]]

## 主题结论
Pretext 不是只有一个库入口文件的轻量 demo 仓库，而是一个“库代码 + 浏览器验证页面 + 自动化校验脚本 + 快照数据 + 研究日志”共同组成的工程系统。它的可信度来自这整套验证面。

## 发布面

### npm 包入口
`package.json` 当前把发布入口明确指向：
- `main: ./dist/layout.js`
- `types: ./dist/layout.d.ts`
- `exports["."].import/default/types` 同样指向 `dist/`

这和 `CHANGELOG.md` 中 `0.0.3 - 2026-03-29` 的变更一致：
- 从暴露原始 TypeScript 源码
- 切到发布构建后的 ESM + declaration files

### 发布内容
`files` 字段包含：
- `dist`
- `src`，但排除了 `src/layout.test.ts` 与 `src/test-data.ts`
- `pages/demos`
- `pages/assets`
- `CHANGELOG.md`
- `LICENSE`

这里有两个值得注意的点：
- 运行时入口已经是 `dist/`
- 但仓库仍把 `src/` 和 demo/assets 一起纳入发布包，说明“源码可见 + demos 可分发”仍是包体验的一部分

## 构建与类型策略

### 开发时 TypeScript 配置
`tsconfig.json` 的主要特征：
- `module: "Preserve"`
- `moduleResolution: "bundler"`
- `allowImportingTsExtensions: true`
- `noEmit: true`
- 严格模式与一组偏激进的静态约束

这说明开发态主要服务 Bun / bundler / typecheck，而不是直接产出运行时代码。

### 发布时 TypeScript 配置
`tsconfig.build.json` 在开发配置上覆盖为：
- `noEmit: false`
- `allowImportingTsExtensions: false`
- `types: []`
- `rootDir: ./src`
- `outDir: ./dist`
- `declaration: true`

这对应仓库中的一个重要实现约束：
- library source 内部导入要写 `.js` specifier
- 这样 plain `tsc` 才能直接产出运行时正确的 JS 与 `.d.ts`

## 命令面
从 `DEVELOPMENT.md` 与 `package.json` 可以把命令分为五类：

### 1. 开发与站点
- `bun start`
- `bun run start:lan`
- `bun run start:watch`
- `bun run site:build`

### 2. 静态质量
- `bun run check`
- `bun test`

### 3. 发布与包校验
- `bun run build:package`
- `bun run package-smoke-test`
- `prepack`

### 4. 浏览器准确性/性能校验
- `bun run accuracy-check*`
- `bun run benchmark-check*`
- `bun run pre-wrap-check`

### 5. 语料/Gatsby/诊断
- `bun run corpus-*`
- `bun run gatsby-*`
- `bun run probe-check*`

## Source Of Truth 设计
仓库把“当前结论”和“历史推导”刻意分开：
- 顶层 `STATUS.md`：当前浏览器准确性与 benchmark 快照
- `accuracy/*.json` / `benchmarks/*.json`：机器可读快照
- `corpora/STATUS.md` / `corpora/representative.json`：长文 canary 当前状态
- `RESEARCH.md`：为什么变成现在这样
- `TODO.md`：当前优先级与明确不该做的事

这套分层很有价值，因为它避免了一个文件既想做 dashboard、又想做历史日志、又想做路线图。

## 工程约束的产品含义
- 库的正确性主张依赖浏览器页面与脚本，不是只靠单元测试
- 包发布形态最近刚发生重要变化，说明工程表面还在演化
- `package-smoke-test` 被单独保留，表明“发布产物正确”被当成独立风险，而不是顺带检查
- `pre-wrap` 有单独永久 oracle，说明仓库倾向于为新增行为建立小而稳定的 standing check

## 初步评价

### 值得借鉴
- 发布面、开发面、验证面边界清楚
- 当前 source of truth 有明确分层
- 发布构建与包烟测被纳入正式脚本，而不是靠人工记忆

### 后续研究点
- 继续追踪 `pages/` 与 `scripts/` 如何把这些工程承诺落实成可重复流程
- 继续核对 `STATUS.md` / `RESEARCH.md` / `TODO.md` 中的说法在代码里分别落到哪里
