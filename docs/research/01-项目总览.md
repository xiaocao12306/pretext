# Pretext 项目总览

Pretext 在这个仓库里表现为一个“浏览器字体度量驱动的、多语言、多行文本布局预测库”。它的目标不是自己成为完整排版引擎，而是在不触发 DOM reflow 的前提下，给应用层一个足够接近浏览器排版结果的高度/行布局预测能力。

## 核心问题
浏览器里直接靠 `getBoundingClientRect()`、`offsetHeight` 之类的 DOM 读来量文本，会把组件局部需求升级成整页同步 layout。Pretext 的中心论点是：
- 冷路径一次性做文本分析和测量
- 热路径只做算术行走
- 把“测量成本”和“resize 频率”拆开

这也是 `prepare()` / `layout()` 分裂的根本原因，详见 [[02-模块/00-模块总览]]。

## 公开能力
从 `README.md` 与 `src/layout.ts` 看，Pretext 有两类主要能力：

### 1. 高度预测
- `prepare(text, font, options?)`
- `layout(prepared, maxWidth, lineHeight)`

这是最强调性能的路径。`prepare()` 做文本分析与 canvas 测量；`layout()` 使用缓存宽度做纯算术断行与高度计算。

### 2. 富行布局
- `prepareWithSegments()`
- `layoutWithLines()`
- `walkLineRanges()`
- `layoutNextLine()`

这一组 API 不改变核心断行规则，但暴露更多结构信息，让调用方可以手工绘制、做 shrink-wrap、做 obstacle-aware layout 或按行流式布局。

## 初步架构图

### 1. 文本分析层
文件：`src/analysis.ts`

职责：
- whitespace 归一化
- `Intl.Segmenter` 分词
- segment break kind 分类
- glue / CJK / Arabic / URL / 数字串 / escaped quote cluster 等预处理
- 为 `pre-wrap` 建 chunk 边界

### 2. 度量层
文件：`src/measurement.ts`

职责：
- 获取 canvas context
- 以 `Map<font, Map<segment, metrics>>` 缓存 segment metrics
- 浏览器 profile 探测
- emoji correction
- breakable segment 的 grapheme 宽度与 prefix 宽度缓存

### 3. 行走层
文件：`src/line-break.ts`

职责：
- 热路径 `layout()` 的纯算术行数计算
- rich path 的 line range 步进
- 处理 soft hyphen、tab、pre-wrap hard break、breakable run、line-fit epsilon

### 4. API 编排层
文件：`src/layout.ts`

职责：
- 对外暴露 `prepare()`、`layout()`、rich line APIs
- 把 analysis / measurement / bidi / line-break 组装成统一 PreparedText 模型
- 保持 opaque fast path 与 rich path 的边界

### 5. Rich-path bidi 辅助层
文件：`src/bidi.ts`

职责：
- 仅在 `prepareWithSegments()` 路径上生成 segment level 元数据
- 不参与真正的行断逻辑

## 代码层面的关键分裂

### 冷路径 vs 热路径
- 冷路径：`analyzeText()` + `measureAnalysis()`，复杂规则和浏览器差异都尽量压在这里
- 热路径：`countPreparedLines()`，追求“无 DOM 读、无 canvas 调用、无字符串工作、少分配”

### opaque handle vs rich handle
- `prepare()` 返回 opaque `PreparedText`
- `prepareWithSegments()` 返回 richer `PreparedTextWithSegments`

这说明仓库明确拒绝把内部并行数组表示直接固化为公共 API，但同时保留实验/手工布局出口。

### 语义规则 vs 浏览器特征
- 语义规则主要在 `src/analysis.ts`
- 浏览器特征与 shim 主要在 `src/measurement.ts`
- 断行执行主要在 `src/line-break.ts`

这种分工很重要，因为它把“为什么这样分段”与“如何按宽度走线”分开了。

## 仓库其余部分如何围绕核心引擎工作
- `pages/` 不是产品包的一部分，而是 demo、accuracy、benchmark、probe、corpus、gatsby 的浏览器验证层
- `scripts/` 把这些页面变成可重复的 checker/snapshot 流程
- `accuracy/`、`benchmarks/`、`corpora/` 提供当前 source-of-truth 快照
- `RESEARCH.md` 保存“为什么现在长这样”的历史研究轨迹

更细的目录分层见 [[01-仓库地图]]。

## 初步设计判断

### 值得借鉴
- 核心 API 非常克制：产品面小，内部演化空间大
- 冷/热路径边界明确，性能意图直接写进实现
- rich line APIs 建在同一 breaker 之上，没有分叉成第二套语义
- 验证体系完整：页面、脚本、JSON 快照、研究日志彼此闭环

### 复杂与脆弱点
- `src/analysis.ts` 规则密集，脚本特化逻辑很多，后续需要逐条解释来源和边界
- `src/line-break.ts` 既服务 hot path 又服务 rich path，虽然已做 fast path 分裂，但复杂度仍高
- 正确性高度依赖浏览器 ground truth，因此 `pages/` 与 `scripts/` 实际上是核心工程的一部分，不是可有可无的 demo

## 当前研究结论
- Pretext 的主产品不是“画文本”，而是“预测文本如何被浏览器画出来”
- 真正的架构中心不是 `layout()` 一支函数，而是 `analysis -> measurement -> prepared arrays -> line break` 这条链
- 这个仓库的研究不能只盯 `src/`，因为大量正确性证据藏在 `pages/`、`scripts/`、快照和研究日志里
