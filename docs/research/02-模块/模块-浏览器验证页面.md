## 模块定位
这一组页面不是简单的可视化 demo，而是 Pretext 的浏览器验证前线。它们共同承担三件事：
- 把库结果与真实 DOM 结果做对比
- 产出可被脚本消费的 machine-readable report
- 为特定 mismatch 提供足够深的现场诊断

## 页面分工

### `pages/accuracy.ts`
作用：
- 运行共享 `TEXTS × SIZES × WIDTHS × FONTS` sweep
- 用 DOM block height 与 `layout()` 结果对比
- 对 mismatch 再用 `layoutWithLines()` 和 DOM line extractor 生成 per-line diagnostics

它对应的是“广覆盖回归页”。

### `pages/benchmark.ts`
作用：
- 测 `prepare()` / `layout()` / DOM batch / DOM interleaved`
- 测 rich line APIs：`layoutWithLines()` / `walkLineRanges()` / `layoutNextLine()`
- 测长文 corpus 的 `analysisMs` / `measureMs` / `prepareMs` / `layoutMs`

它对应的是“性能基线页”。

### `pages/corpus.ts`
作用：
- 针对单个 corpus 做更重的宽度/换行诊断
- 支持 sweep 宽度、多 extractor、slice、font/lineHeight override
- 比较：
  - 预测高度 vs DOM 高度
  - 预测 line breaks vs browser line breaks
  - segment sum width / canvas full-string width / DOM width

它对应的是“长文 canary 与深诊断页”。

### `pages/probe.ts`
作用：
- 针对给定短文本做可参数化 probe
- 比较 range/span 两种 browser line extractor
- 输出 first break mismatch 与 extractor sensitivity

它对应的是“细小复现与局部诊断页”。

## 共享模式

### 1. URL 参数驱动
这些页面普遍通过 query params 控制行为，例如：
- `requestId`
- `reportEndpoint`
- `width` / `widths`
- `font`
- `lineHeight`
- `method`
- `diagnostic`

这说明页面设计从一开始就考虑到自动化脚本消费，而不是只给人手点。

### 2. 导航态报告
所有页面都通过 `report-utils.ts` 更新 hash：
- `loading`
- `measuring`
- `posting`
- ready/error report

这让脚本即使不直接注入页面上下文，也能从导航状态知道页面走到哪一步。

### 3. 大小报告的双通道
- 小报告：hash 导航态
- 大报告：本地 `reportEndpoint` POST side channel

这在 `accuracy.ts` 与 `corpus.ts` 都是显式存在的。

### 4. 诊断辅助
`diagnostic-utils.ts` 提供：
- grapheme-safe diagnostic units
- line content trimming
- break context formatting
- canvas / DOM 单段宽度测量

也就是说，页面层没有各写各的 offset/width helper，而是共享同一组诊断原语。

## 与核心引擎的关系

### 使用 fast path
- `accuracy.ts`、`benchmark.ts` 大量调用 `layout()`

### 使用 rich path
- `accuracy.ts` mismatch 诊断时调用 `layoutWithLines()`
- `benchmark.ts` 专门 benchmark rich APIs
- `corpus.ts`、`probe.ts` 深度依赖 `prepareWithSegments()` 与 `layoutWithLines()`

这让页面层同时验证：
- fast path 的预测能力
- rich path 的几何/文本对齐能力

## 当前判断

### 值得借鉴
- 页面天然可脚本化，自动化接口不是事后补上的
- 从 broad sweep 到 deep probe 的梯度非常清楚
- hash phase + POST side channel 的组合很适合长报告与超时诊断

### 风险点
- 页面层逻辑已经很重，尤其 `corpus.ts` 与 `probe.ts`，后续要留意是否应进一步下沉共享诊断逻辑
