## 模块定位
这两份文件共同体现了仓库对“持久测试”和“浏览器验证数据”的分层：
- `src/layout.test.ts`：小而稳定的 shipped API invariant suite
- `src/test-data.ts`：被浏览器 accuracy/benchmark 页面与 checker 共用的数据集

它们不是同一种东西，仓库刻意把这两类验证分开。

## `src/layout.test.ts`

### 目标
文件头已经写得很明确：
- 永久测试保持 small and durable
- 用 deterministic fake canvas backend
- 窄而浏览器特定的调查，优先走 throwaway probes 和 browser checkers

这解释了为什么这里虽然有 600+ 行，但仍是“不变量仓库”，不是浏览器精度镜像。

### 测试环境
- 用 `bun:test`
- 在 `beforeAll` 里注入 `TestOffscreenCanvas`
- `measureText()` 由本地 `measureWidth()` 决定

这意味着测试的目的不是复刻真实浏览器排版，而是给 exported APIs 一个稳定、可控、无外部波动的度量环境。

### 覆盖面
测试大体分两类：

#### 1. `prepare invariants`
覆盖：
- `normal` / `pre-wrap` whitespace 语义
- `NBSP` / `NNBSP` / `WJ` / `ZWSP` / `SHY`
- Arabic / Devanagari / Myanmar / CJK / astral CJK
- quotes / escaped quotes / URL / punctuation chains / numeric ranges
- `prepare()` 与 `prepareWithSegments()` 行为一致性
- `setLocale()` 的缓存与后续 prepare 关系

#### 2. `layout invariants`
覆盖：
- line count 单调性
- trailing whitespace hanging
- breakable word 的 grapheme 级断行
- mixed bidi smoke coverage
- `layoutNextLine()` 与 `layoutWithLines()` 对齐
- `pre-wrap` 的 spaces / tabs / hard breaks / empty lines / trailing-final-break
- `walkLineRanges()` 与 `layoutWithLines()` 几何一致
- `countPreparedLines()` 与 `walkPreparedLines()` 计数一致

### 设计判断
这份测试的重点是：
- 检查 exported API 与 rich path/fast path 的语义一致性
- 检查几个最易回归的 durable invariants
- 不把整套浏览器差异快照搬进单元测试

## `src/test-data.ts`

### 目标
它是 browser accuracy pages/checkers 和 benchmark page 的共享输入语料，不是单元测试断言文件。

### 数据组成
- `TEXTS`：30 条左右共享文本，覆盖 Latin、Arabic、Hebrew、mixed bidi、CJK、Korean、Thai、emoji、mixed everything、空串、空白、换行、超长词
- `SIZES`：8 个字号
- `WIDTHS`：8 个容器宽度

结合顶层状态文档可以推断：
- 这些共享维度被页面与脚本扫成更大的浏览器准确性矩阵
- 例如 4 font × 8 size × 8 width × 30 text = 7680 条浏览器 sweep 行

### 设计作用
这份文件的价值不只是“省得重复写样本文本”，而是：
- 保持 accuracy 与 benchmark 页面共享同一批代表性输入
- 让性能讨论和准确性讨论至少有一部分落在同一语料分布上

## `src/text-modules.d.ts`
这份文件非常小，但它补上了一个实际工程缝隙：
- 允许 TypeScript 把 `*.txt` 语料文件视为 `string` 模块导入

它的存在说明语料文本不只是静态资产，也会直接进入页面/脚本代码路径。

## 当前判断

### 值得借鉴
- durable unit tests 与 browser-oracle 验证被清楚分层
- 共享数据集既支持性能页，也支持准确性页，减少了“各测各的”语义漂移

### 风险点
- `src/layout.test.ts` 已经很长，后续若继续把 browser-specific case 塞进来，会偏离“小而稳定”的原设计
- `src/test-data.ts` 作为共享语料，如果修改过多，会同时影响 benchmark 与 accuracy 的可比性
