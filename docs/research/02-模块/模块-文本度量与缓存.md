# 模块：文本度量与缓存

## 模块定位
`src/measurement.ts` 是 `prepare()` 冷路径的第二段。它负责把 `analysis.ts` 给出的 segment 变成可复用的宽度事实，并把浏览器差异控制在热路径之外。

## 核心职责
- 提供 canvas measure context
- 维护 segment metrics cache
- 解析 font size
- 探测浏览器 engine profile
- 处理 emoji canvas/DOM 差异
- 为 breakable run 准备 grapheme widths 与 prefix widths

## 核心数据结构

### `SegmentMetrics`
字段包括：
- `width`
- `containsCJK`
- `emojiCount?`
- `graphemeWidths?`
- `graphemePrefixWidths?`

这个结构不是只缓存一个 width，而是按需扩展：
- 先缓存最便宜、最通用的结果
- 只有需要时才补 emoji 计数或 grapheme 级宽度

### 缓存层级
`segmentMetricCaches` 的结构是：
- `Map<font, Map<segment, SegmentMetrics>>`

这很关键，因为它表明缓存是“同字体共享、跨文本复用”的，而不是绑在某个 prepared handle 上。

## 核心机制

### 1. canvas context 获取
`getMeasureContext()` 的优先级是：
- `OffscreenCanvas`
- DOM canvas
- 两者都没有则抛错

这说明当前仓库的度量能力明确锚定在浏览器或浏览器兼容环境。

### 2. 浏览器 profile 探测
`getEngineProfile()` 目前根据 UA/vendor 派生：
- `lineFitEpsilon`
- `carryCJKAfterClosingQuote`
- `preferPrefixWidthsForBreakableRuns`
- `preferEarlySoftHyphenBreak`

这里有两个重要边界：
- 它只暴露少量确实需要的 engine knobs
- profile 的结果主要被 `analysis.ts` 与 `line-break.ts` 消费，而不是直接污染公共 API

### 3. emoji correction
这是本模块最典型的“浏览器差异隔离”案例：
- 先用 canvas 测 emoji
- 必要时再用隐藏 DOM span 读取实际宽度
- 如果 canvas 比 DOM 宽出可观差值，就缓存 correction

结果是：
- 差异探测发生在冷路径
- correction 以 font 为键缓存
- 热路径只消费已经修正过的 segment width

### 4. breakable run 的 grapheme 宽度
`getSegmentGraphemeWidths()` 与 `getSegmentGraphemePrefixWidths()` 只在需要时生成：
- `graphemeWidths` 主要服务字符级断行
- `graphemePrefixWidths` 主要服务 Safari 一类需要 prefix 宽度判定的场景

这说明仓库没有默认把所有 segment 都拆到 grapheme 级，而是按需付费。

## 与其他模块的协作

### 与 `analysis.ts`
- `analysis.ts` 决定 segment 语义
- `measurement.ts` 决定这些 segment 的宽度事实和浏览器 shim

### 与 `layout.ts`
`layout.ts` 的 `measureAnalysis()` 是 measurement 真正的编排调用点。它负责：
- 拿到 font measurement state
- 计算 discretionary hyphen width
- 计算 tab stop advance
- 决定是否要给某个 segment 准备 breakable widths

### 与 `line-break.ts`
`measurement.ts` 不参与实际断行，但它提供了断行所需的关键输入：
- `breakableWidths`
- `breakablePrefixWidths`
- `lineFitEpsilon`
- soft-hyphen 与 tab 相关宽度参数

## 设计判断

### 值得借鉴
- 缓存与 shim 都被安排在冷路径
- 浏览器差异被集中表达为少量 profile/correction，而不是散落在各处 if-else
- segment metrics 按需增肥，不一次性缓存所有导出事实

### 风险点
- UA/vendor 判定天然有脆弱性，后续需要结合页面/脚本验证其维护成本
- `SegmentMetrics` 已经承担多种派生事实，后续需要注意不要演变成无边界杂项袋
