## 模块定位
`src/analysis.ts` 是 `prepare()` 冷路径的第一段。它的职责不是测量宽度，而是把原始文本变成“带断行语义的 segment 序列”。

这里发生的工作包括：
- white-space 归一化
- `Intl.Segmenter` 词级分段
- segment break kind 分类
- 特定脚本/场景的预处理与合并
- `pre-wrap` 模式的 hard-break chunk 编译

## 输入与输出

### 输入
- 原始 `text`
- `AnalysisProfile`
  - 当前主要来自 `measurement.ts` 的浏览器 profile
  - 例如 `carryCJKAfterClosingQuote`
- `whiteSpace`
  - `'normal'`
  - `'pre-wrap'`

### 输出
`TextAnalysis`，其核心内容是：
- `normalized`
- `texts`
- `isWordLike`
- `kinds`
- `starts`
- `chunks`

这是一套并行数组结构，后续由 `src/layout.ts` 的 `measureAnalysis()` 消费。

## 关键数据模型

### `SegmentBreakKind`
当前至少区分 8 类：
- `text`
- `space`
- `preserved-space`
- `tab`
- `glue`
- `zero-width-break`
- `soft-hyphen`
- `hard-break`

这说明 Pretext 没有把“可断 / 不可断”粗暴压成一个布尔值，而是保留了后续 layout 所需的细粒度语义。

### `WhiteSpaceMode`
- `normal`
- `pre-wrap`

这里的 `pre-wrap` 是一个刻意收窄的工程实现：
- 保留 ordinary spaces
- 保留 tabs
- 保留 hard breaks
- 其他断行默认仍保持 `word-break: normal`、`overflow-wrap: break-word` 一类的现有目标

## 分析流水线

### 1. whitespace 归一化
- `normalizeWhitespaceNormal()`
- `normalizeWhitespacePreWrap()`

`normal` 模式会折叠空白并裁掉首尾普通空格。
`pre-wrap` 模式则保留普通空格和 `\n`，同时把 `\r\n` / `\r` / `\f` 统一到 `\n`。

### 2. 基于 `Intl.Segmenter` 的初始分词
`buildMergedSegmentation()` 以共享的 word segmenter 为入口，把浏览器原生分词作为第一手依据，而不是自己重写整套 Unicode 分词。

### 3. 按 break kind 二次切分
`splitSegmentByBreakKind()` 会把 segment 再拆成不同 break kind 的 piece，例如：
- 文本
- 空格
- `NBSP` / `NNBSP` / `WJ` 这类 glue
- `ZWSP`
- `SHY`
- tab / hard break

### 4. 规则化合并与重写
这是本模块复杂度最高的部分。当前至少包括：
- CJK 行首禁则与 closing-quote carry
- Myanmar medial glue
- Arabic 无空格标点簇
- 标点左黏连与 forward-sticky cluster
- escaped quote cluster
- URL-like run 合并
- URL query run 重新建边界
- 数字串合并
- ASCII punctuation chain 合并
- 连字符数字串再切分
- glue-connected text run 合并
- `" " + combining marks + Arabic` 的特殊处理

这部分的总体策略不是“断行时再去猜规则”，而是尽可能把脚本特性前移到预处理层。

### 5. chunk 编译
`compileAnalysisChunks()` 负责把 `pre-wrap` hard break 编译成 chunk。后续 `line-break.ts` 就可以按 chunk 行走，而不必每次重新理解 hard-break 语义。

## 设计意图

### 1. 把脚本特化留在预处理层
从函数布局看，仓库明显不希望在 hot `layout()` 路径里塞入 Arabic/CJK/URL 之类的特殊政策。复杂度被前移到了 `prepare()`。

### 2. 让 segment 语义足够富
`kinds`、`isWordLike`、`starts`、`chunks` 共同构成了后续测量与断行的语义基础。也因此，这个模块实际上决定了“后面还能不能高效”。

### 3. 使用浏览器能力，但不完全依赖它
`Intl.Segmenter` 是基础，但仓库并不把它当成终点；大量后处理逻辑说明“浏览器原生分词”与“浏览器最终排版行为”之间仍有语义缝隙。

## 当前判断

### 值得借鉴
- 明确把语言/脚本规则集中在一个模块，不污染 hot path
- 通过 `SegmentBreakKind` 保留足够细的下游语义
- URL、数字串、escaped quote 这类“产品文本形状”被认真建模，不只盯文学语料

### 风险点
- 规则数量已经很多，后续要继续记录“哪条规则来自哪类 canary / 研究结论”
- 各 merge pass 的顺序本身就是语义的一部分，后续实现卡需要特别解释这个顺序依赖
