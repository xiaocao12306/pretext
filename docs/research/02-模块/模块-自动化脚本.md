## 模块定位
自动化脚本层的职责不是重新实现页面逻辑，而是：
- 启动页面
- 驱动浏览器加载目标页面
- 等待页面通过 hash 或 POST 回传 machine-readable report
- 将页面结果变成 CLI / JSON snapshot 输出

## 核心桥接：`browser-automation.ts`

### `BrowserSession`
对不同浏览器统一出最小接口：
- `navigate(url)`
- `readLocationUrl()`
- `close()`

具体实现：
- Chrome：AppleScript 控制标签页
- Safari：AppleScript 控制窗口/标签
- Firefox：临时 profile + BiDi WebSocket

这说明自动化层追求的是“够用的统一页面驱动”，不是抽象出完整浏览器自动化框架。

### 自动化锁
`acquireBrowserAutomationLock()` 用临时文件保证：
- 同一浏览器一次只跑一个 checker
- stale dead-owner lock 能自愈清理

这和仓库文档里“不并行跑同浏览器 sweep job”的约束是一致的。

### 页面服务器
`ensurePageServer()`：
- 若目标页已可访问，则复用现有 Bun server
- 否则起临时 `bun --no-hmr pages/*.html`

这让 checker 能独立运行，同时尽量复用本地现有页面服务。

### 报告读取
- `loadHashReport()`：轮询 URL hash 中的 report
- `loadPostedReport()`：等待本地 report server 的 POST，同时持续观察最后 phase

二者都用 `requestId` 过滤本次运行的报告。

## 报告通道

### `shared/navigation-state.ts`
定义 hash 协议：
- phase：`loading` / `measuring` / `posting`
- report：JSON string

### `scripts/report-server.ts`
提供本地临时 HTTP server：
- 接收页面 POST 的大报告
- 按 `requestId` 解析/等待
- 给 checker 提供 `waitForReport()`

## 首批 checker

### `scripts/accuracy-check.ts`
职责：
- 启动 `/accuracy`
- 根据 `--full` 决定走 hash 还是 POST side channel
- 输出 mismatch 摘要，必要时写 JSON

特别点：
- Firefox 路径通过本地 proxy 解决访问问题

### `scripts/benchmark-check.ts`
职责：
- 启动 `/benchmark`
- 读取 benchmark report
- 输出控制台摘要或写 JSON

特别点：
- 只支持 Chrome / Safari
- session 默认 `foreground: true`，与仓库 benchmark 前台运行原则一致

### `scripts/corpus-check.ts`
职责：
- 启动 `/corpus`
- 传入宽度、font、lineHeight、method、slice 等 override
- 打印 break mismatch、width drift、extractor sensitivity 等细诊断信息

## 当前判断

### 值得借鉴
- 页面与脚本之间只有一个很小的协议面：phase + report + requestId
- 浏览器驱动、页面服务、报告传输被拆成独立职责
- checker 本身尽量薄，复杂诊断仍留在页面侧

### 风险点
- AppleScript 和 Firefox BiDi 路径具有平台依赖性，后续需要留意维护成本
- 自动化层与页面 hash/POST 协议是强耦合关系，若页面协议变化，脚本会一起受影响
