## 模块定位
自动化脚本层的职责不是重新实现页面逻辑，而是：
- 启动页面
- 驱动浏览器加载目标页面
- 等待页面通过 hash 或 POST 回传 machine-readable report
- 将页面结果变成 CLI / JSON snapshot 输出

## 核心桥接：`browser-automation.ts`

### `BrowserSession`
对不同浏览器统一出最小接口：
- `navigate(url)`
- `readLocationUrl()`
- `close()`

具体实现：
- Chrome：AppleScript 控制标签页
- Safari：AppleScript 控制窗口/标签
- Firefox：临时 profile + BiDi WebSocket

这说明自动化层追求的是“够用的统一页面驱动”，不是抽象出完整浏览器自动化框架。

### 自动化锁
`acquireBrowserAutomationLock()` 用临时文件保证：
- 同一浏览器一次只跑一个 checker
- stale dead-owner lock 能自愈清理

这和仓库文档里“不并行跑同浏览器 sweep job”的约束是一致的。

### 页面服务器
`ensurePageServer()`：
- 若目标页已可访问，则复用现有 Bun server
- 否则起临时 `bun --no-hmr pages/*.html`

这让 checker 能独立运行，同时尽量复用本地现有页面服务。

### 报告读取
- `loadHashReport()`：轮询 URL hash 中的 report
- `loadPostedReport()`：等待本地 report server 的 POST，同时持续观察最后 phase

二者都用 `requestId` 过滤本次运行的报告。

## 报告通道

### `shared/navigation-state.ts`
定义 hash 协议：
- phase：`loading` / `measuring` / `posting`
- report：JSON string

### `scripts/report-server.ts`
提供本地临时 HTTP server：
- 接收页面 POST 的大报告
- 按 `requestId` 解析/等待
- 给 checker 提供 `waitForReport()`

## 首批 checker

### `scripts/accuracy-check.ts`
职责：
- 启动 `/accuracy`
- 根据 `--full` 决定走 hash 还是 POST side channel
- 输出 mismatch 摘要，必要时写 JSON

特别点：
- Firefox 路径通过本地 proxy 解决访问问题

### `scripts/benchmark-check.ts`
职责：
- 启动 `/benchmark`
- 读取 benchmark report
- 输出控制台摘要或写 JSON

特别点：
- 只支持 Chrome / Safari
- session 默认 `foreground: true`，与仓库 benchmark 前台运行原则一致

### `scripts/corpus-check.ts`
职责：
- 启动 `/corpus`
- 传入宽度、font、lineHeight、method、slice 等 override
- 打印 break mismatch、width drift、extractor sensitivity 等细诊断信息

## 第二批脚本：批量 sweep 与研究沉淀

### `scripts/gatsby-check.ts` vs `scripts/gatsby-sweep.ts`
- `gatsby-check.ts` 是慢速、重诊断的单点工具，按宽度逐个跑 `/gatsby`
- `gatsby-sweep.ts` 是 batched 宽度 sweep，会把 mismatch 宽度分 bucket，总结后还能可选回退到 `gatsby-check.ts`

这是一个很明确的二级工具链：
- sweep 负责筛
- check 负责解释

### `scripts/corpus-sweep.ts`
- 面向一个或全部 corpus 做 coarse sweep
- 从 `corpora/sources.json` 读取 corpus metadata
- 自动尊重每个 corpus 的 `min_width` / `max_width`
- 支持 `samples`、`font`、`lineHeight` override

### `scripts/corpus-font-matrix.ts`
- 不是所有 corpus 都跑字体矩阵，而是只对少量重点 corpus 配置 `FONT_MATRIX`
- 每个语种/语料配置一组真实字体组合
- 用来回答“同一 corpus 在不同系统字体/回退链下还能不能保持健康”

### `scripts/corpus-taxonomy.ts`
- 不只是打印 mismatch，而是按启发式规则归类成 taxonomy
- 这层逻辑已经是研究辅助，而不只是 checker 包装

### `scripts/corpus-representative.ts`
- 生成 `corpora/representative.json`
- 固定 corpus 集、固定少量宽度、固定浏览器集
- 额外保存浏览器环境指纹

这说明自动化层已经开始主动生产长期研究资产，而不是只做一次性 CLI 输出。

## 专项 checker 与非浏览器脚本

### `scripts/pre-wrap-check.ts`
- 用一小组永久 oracle case 守护 `{ whiteSpace: 'pre-wrap' }`
- case 覆盖 preserved spaces、tabs、hard breaks、mixed-script indent、RTL indent
- 当前 CLI 接受 `firefox` 标记，但实现上仍明确报“不支持 Firefox pre-wrap oracle”

### `scripts/probe-check.ts`
- 是 `/probe` 的通用 CLI 包装
- 适合最小复现、extractor 对比、`dir/lang/whiteSpace` 试验

### `scripts/package-smoke-test.ts`
- 面向 tarball，而不是工作区源码
- 通过临时 JS/TS consumer 验证发布包的 runtime/export/type surface

### `scripts/build-demo-site.ts`
- 构建 GitHub Pages 静态 demo 站点
- 额外处理 HTML 位置迁移、相对资源路径 rebasing、landing page 链接改写

## 当前判断

### 值得借鉴
- 页面与脚本之间只有一个很小的协议面：phase + report + requestId
- 浏览器驱动、页面服务、报告传输被拆成独立职责
- checker 本身尽量薄，复杂诊断仍留在页面侧
- batched sweep + slow diagnose + snapshot generation 已经形成稳定分层
- 自动化脚本并不都依赖浏览器；tarball smoke test 与 demo-site build 把发布面也纳入了脚本治理

### 风险点
- AppleScript 和 Firefox BiDi 路径具有平台依赖性，后续需要留意维护成本
- 自动化层与页面 hash/POST 协议是强耦合关系，若页面协议变化，脚本会一起受影响
