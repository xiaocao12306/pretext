## 模块定位
这一层不是代码实现本身，而是 Pretext 研究与工程主张背后的“可追溯证据资产”。

它包含四类东西：
- browser accuracy 原始快照
- benchmark 原始快照
- corpus 元数据与长文 canary 文本
- 面向人读的 compact dashboard 与 mismatch 词汇表

如果没有这一层，`pages/` 和 `scripts/` 只会产生短暂报告；有了这一层，仓库才能把“当前状态”和“长期研究资产”区分开来。

## 资产分层

### 1. 浏览器准确性快照：`accuracy/*.json`
结构特征：
- 顶层有 `status`
- 带完整 `environment`
- 有 `total` / `matchCount` / `mismatchCount`
- 保留 `mismatches`
- 同时保留全量 `rows`

这说明 accuracy 快照不是只保存一个总分，而是同时服务：
- dashboard 汇总
- 历史比较
- 需要时回看具体 row

从当前快照看，三浏览器都已经到 `7680/7680`。

### 2. benchmark 快照：`benchmarks/*.json`
结构特征：
- 分成 `results`、`richResults`、`richLongResults`、`corpusResults`
- 同时记录微基准和长文基准
- `corpusResults` 不只是时间，还保留：
  - `chars`
  - `analysisSegments`
  - `segments`
  - `breakableSegments`
  - `lineCount`
  - `analysisMs` / `measureMs` / `prepareMs` / `layoutMs`

这意味着 benchmark 快照关心的不只是“快不快”，还关心“是什么文本结构导致这个成本”。

### 3. corpus 元数据：`corpora/sources.json`
这里是 corpus 的机器可读注册表。

每条记录至少含：
- `id`
- `language`
- `direction`
- `title`
- `source_url`
- `output`
- `font_family`
- `font_size_px`
- `line_height_px`
- `default_width`
- `min_width`
- `max_width`
- `characters`
- `lines`
- `acquired_via`
- `notes`

它的作用不是简单索引文件名，而是同时固化：
- canonical corpus identity
- canonical font/lineHeight
- 合理 sweep 范围
- 采集来源与清洗方式

### 4. corpus 文本本体：`corpora/*.txt`
这些文本不是“随便找几篇文章”。

从 `corpora/README.md` 和 `sources.json` 看，它们是明确筛过的 canary：
- 有真实长文语言覆盖
- 有 source provenance
- 有清洗说明
- 明确知道哪些是 synthetic，哪些来自 Wikisource

它们和 `src/test-data.ts` 的关系很重要：
- `src/test-data.ts` 更像共享 sweep / benchmark 样本
- `corpora/*.txt` 是更接近真实产品形状的长文 stress canary

### 5. corpus compact dashboard：`corpora/STATUS.md`
这是面向人读的压缩状态页。

它做了三件事：
- 把结果按 Browser Regression Gate / Product-Shaped Canary / Long-Form Corpora / Fine-Sweep / Font Matrix 分层
- 把“当前值得记住的 field”压缩成短说明
- 把 recompute 命令直接和当前状态绑在一起

`corpora/STATUS.md` 明显不是原始数据文件，而是运营面板。

### 6. mismatch 词汇表：`corpora/TAXONOMY.md`
这里不是记录结果，而是记录解释结果的共享 vocabulary。

当前分类包括：
- `corpus-dirty`
- `normalization`
- `boundary-discovery`
- `glue-policy`
- `edge-fit`
- `shaping-context`
- `font-mismatch`
- `diagnostic-sensitivity`

这使 corpus 研究不再只是“哪个宽度错了”，而是“这个错属于哪一类工程动作”。

### 7. representative snapshot：`corpora/representative.json`
这是另一种压缩方式：
- 比 `corpus/STATUS.md` 更机器可读
- 比完整 `corpus-sweep` 结果更小、更稳定

它当前固定：
- 一组 `corpora`
- 一组 `widths`
- 每个 browser 的 `environment`
- 每个代表 row 的预测/实测结果

这类文件适合长期版本追踪，而不适合临时诊断。

### 8. demo 资产文件：`pages/assets/*`
`openai-symbol.svg` / `claude-symbol.svg` 虽然不在 `corpora/` 下，但在研究体系中的角色和普通静态资源不同。

它们是：
- demo 展示素材
- 也是 `wrap-geometry.ts` 的几何输入

所以这类资产和 corpus 文本一样，都属于“被布局系统消费的外部输入”，而不只是装饰资源。

## 当前判断

### 值得借鉴
- 仓库把原始快照、compact dashboard、代表样本、taxonomy 词汇表拆成了不同文件，而不是混成一个万能状态页
- `sources.json` 让 corpus 从“文件集合”升级成“带 canonical 参数和 provenance 的注册表”
- benchmark 快照保留结构统计，便于把性能结果和文本形态联系起来

### 风险点
- `accuracy/*.json` 和 `benchmarks/*.json` 的体量会继续增长，后续要留意哪些字段必须长期保留，哪些只是阶段性诊断冗余
- `corpora/STATUS.md` 是人工压缩页，若与 raw snapshot 长期不同步，就会损伤这套证据链的可信度
