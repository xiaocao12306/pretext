## 模块定位
`src/line-break.ts` 是 Pretext 真正执行“按宽度走行”的核心模块。它不负责文本预处理，也不负责 canvas 测量，而是消费 prepared arrays，把断行语义落实为：
- 行数统计
- 行范围遍历
- 流式下一行步进

## 输入模型
断行引擎只接收 `PreparedLineBreakData`，其中最关键的字段包括：
- `widths`
- `lineEndFitAdvances`
- `lineEndPaintAdvances`
- `kinds`
- `simpleLineWalkFastPath`
- `breakableWidths`
- `breakablePrefixWidths`
- `discretionaryHyphenWidth`
- `tabStopAdvance`
- `chunks`

这意味着 line breaker 的输入已经是高度预处理过的“断行事实表”，不是原始文本。

## 对外能力

### 1. `countPreparedLines()`
服务 hot `layout()`。

特点：
- 只返回行数
- 尽量不做额外 bookkeeping
- 在 `simpleLineWalkFastPath` 为真时走简化路径

### 2. `walkPreparedLines()`
服务 `layoutWithLines()` 与 `walkLineRanges()`。

特点：
- 返回每行的 `start/end` cursor 与宽度
- 保留 rich path 需要的范围信息

### 3. `layoutNextLineRange()`
服务 `layoutNextLine()`。

特点：
- 从任意 cursor 开始流式取下一行
- 支持不同调用间动态改变宽度

### 4. `normalizeLineStart()`
是 rich streaming path 的关键辅助：
- 跳过 chunk 开头不可见的 hanging 空白或 discretionary-only 边界
- 处理 hard-break chunk 的起点归一化

## 核心结构：双路径

### Simple fast path
入口：
- `countPreparedLinesSimple()`
- `walkPreparedLinesSimple()`
- `layoutNextLineRangeSimple()`

适用条件来自 `layout.ts` 的 prepared 构建阶段：只有普通 `text` / `space` / `zero-width-break` 一类形态时，才允许走这条路。

特点：
- 无需处理 `tab`、`hard-break`、`soft-hyphen` 这些额外状态
- 逻辑更短、更便宜
- 仍支持 breakable segment 的 grapheme 级回退

### General path
入口：
- `walkPreparedLines()`
- `layoutNextLineRange()`

特点：
- 处理 chunk、tab、soft hyphen、fit/paint width 分离、pending break kind
- 同时服务 batched 与 streaming rich path

这两条路径体现了一个重要工程选择：不是所有文本都强行走通用状态机，能快就快，但语义复杂时仍回到统一的全功能路径。

## 核心机制

### 1. pending break
通用路径维护：
- `pendingBreakSegmentIndex`
- `pendingBreakFitWidth`
- `pendingBreakPaintWidth`
- `pendingBreakKind`

这套状态把“哪里可以断”和“断了以后这行算多宽”分开表达。

这对几个行为都很关键：
- trailing spaces hang past line edge
- tab 在 line fit 与 paint 上的差异
- soft hyphen 断点的可见 `-`

### 2. fit width vs paint width
line breaker 不只是问“当前累计宽度是多少”，还区分：
- fit width：判断这一行是否还能放下
- paint width：真正画出来时这一行看起来多宽

这解释了为什么 prepared model 要单独保存 `lineEndFitAdvances` 与 `lineEndPaintAdvances`。

### 3. breakable segments
当单个 segment 太宽时：
- 不回到原始文本重新切
- 直接消费 `breakableWidths` / `breakablePrefixWidths`
- 在 grapheme 级别继续行走

这使 `overflow-wrap: break-word` 类行为仍留在算术域内。

### 4. soft hyphen
`soft-hyphen` 是通用路径的重点复杂度之一：
- 平时自身宽度为 0
- 成为断点时补 `discretionaryHyphenWidth`
- 若后续 breakable segment 还能装下一部分 grapheme，则允许“继续到下一段的一部分再断”

这就是测试里 `trans-a / tlantic` 这类结果能成立的原因。

### 5. tab
tab 不预先记死宽度，而是根据当前 `lineW` 与 `tabStopAdvance` 用 `getTabAdvance()` 动态计算。

因此：
- 同一个 tab segment 在不同行首/行中会有不同 advance
- hard break 后 tab stop 会从新行重新开始

### 6. chunks
`chunks` 主要服务 `pre-wrap` hard break：
- batched walker 按 chunk 循环
- 空 chunk 会直接发出空行
- 最后一个 hard break 不会凭空制造额外 trailing empty line

## 设计判断

### 值得借鉴
- hot path 与 rich path 虽共享语义，但不强迫同一层级的开销
- `pendingBreak*` 状态把复杂行为集中在一个机制里，而不是到处 scattered special cases
- chunk / tab / soft hyphen / breakable run 都在同一状态机里统一落地

### 风险点
- `walkPreparedLines()` 与 `layoutNextLineRange()` 拥有高度相似的状态机分支，后续需要持续关注语义漂移风险
- simple/general path 的双份逻辑有维护成本，但目前这是性能与复杂度之间的现实折中
