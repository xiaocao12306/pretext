# 实现：导航状态与报告通道

相关模块：[[模块-浏览器验证页面]]、[[模块-自动化脚本]]

## 目标
解释页面如何把测量状态与最终结果暴露给自动化脚本，以及为什么仓库需要 hash 与 POST 两条报告通道。

## 协议基础

### `shared/navigation-state.ts`
定义两类 hash 内容：
- phase hash：`#phase=loading|measuring|posting&requestId=...`
- report hash：`#report=<json>`

自动化脚本通过：
- `readNavigationPhaseState()`
- `readNavigationReportText()`
来轮询页面当前状态。

## 页面侧

### phase 发布
页面启动时一般会：
1. `clearNavigationReport()`
2. `publishNavigationPhase('loading', requestId)`
3. 开始测量时 `publishNavigationPhase('measuring', requestId)`
4. 若使用 POST side channel，再在发报告前 `publishNavigationPhase('posting', requestId)`

### report 发布
两种方式：

#### 小报告
直接 `publishNavigationReport(report)`，把 JSON 压进 hash。

#### 大报告
页面检测到 `reportEndpoint` 后：
- `fetch(reportEndpoint, { method: 'POST', body: JSON.stringify(report) })`
- hash 中只保留较小的 navigation report 或 phase

`accuracy.ts`、`corpus.ts`、`gatsby.ts` 都体现了这种分流。

## 脚本侧

### `loadHashReport()`
逻辑：
- 导航到页面 URL
- 轮询当前 URL/hash
- 找到 `report` 后 JSON 解析
- 只接受 `requestId` 匹配的本次结果
- 超时时尽量补读 last phase，给错误信息附带 `loading/measuring/posting`

### `loadPostedReport()`
逻辑：
- 先建本地 `report-server`
- 导航到携带 `reportEndpoint` 的页面 URL
- 并行等待：
  - POST server 收到报告
  - 页面 hash phase 变化
- 若超时，同样带上 last phase 信息

## 为什么需要双通道
- hash 非常适合小报告和纯页面驱动轮询
- 但 sweep rows、full diagnostics 这类大报告不适合塞进 URL/hash
- POST side channel 保持了页面与脚本之间协议简单，同时避免超长 hash

## 当前判断
- 这套通道设计非常工程化：小而够用，没有引入完整 RPC 层
- `requestId` 是关键粘合剂，没有它就无法安全并行或重试读取报告
