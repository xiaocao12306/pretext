# Pretext 研究路线图与检查点

本文档是 [[00-MOC-Pretext]] 的执行骨架，约束后续研究必须沿着“总览 -> 模块 -> 功能 -> 代码实现 -> 专题/风险”的方向推进，而不是零散读文件。

## 研究目标
- 建立 Pretext 的全局认知：它解决什么问题，核心架构怎样分层，主要数据流如何从 `prepare()` 走到 `layout()`
- 按目录和文件研究实现边界：`src/` 核心引擎、`pages/` 浏览器页面、`scripts/` 自动化校验、`corpora/` 语料与快照支撑
- 形成可以互链的知识卡片网络，而不是一篇一次性大报告

## 研究原则
- 每次工作必须能回链到 [[CHECKLIST]] 的某个未完成项
- 每完成一个最小闭环，就在 [[STATUS]] 里记录“完成项 / 下一步 / 风险 / commit 计数”
- 同一子话题的 commit 数量若超过 5，必须在 [[STATUS]] 中记录死循环风险，解释卡点，并切换策略
- 最终每个代码文件都要被纳入某个模块卡、功能卡或实现卡的研究范围

## Roadmap

### Phase 0. 研究脚手架
- 范围：`TASK.md`、[[CHECKLIST]]、[[STATUS]]、[[00-MOC-Pretext]]
- 输出：研究路线、checkpoint、目录导航、第一版总览
- 退出条件：能明确回答“接下来按什么顺序读仓库”

### Phase 1. 仓库与公共表面
- 范围：`README.md`、`DEVELOPMENT.md`、`package.json`、`tsconfig*.json`、`CHANGELOG.md`、`TODO.md`、顶层 `STATUS.md`、`RESEARCH.md`
- 输出：[[01-项目总览]]、[[01-仓库地图]] 的稳定版；公共 API、构建/校验命令、性能/准确性数据源说明
- 退出条件：能说明仓库如何运行、如何发布、如何验证正确性

### Phase 2. 核心引擎模块
- 范围：`src/analysis.ts`、`src/measurement.ts`、`src/line-break.ts`、`src/layout.ts`、`src/bidi.ts`、测试与共享测试数据
- 输出：`docs/research/02-模块/` 下的核心模块卡；`docs/research/04-代码实现/` 下的关键调用链卡
- 退出条件：能说明 `prepare()` / `prepareWithSegments()` / `layout()` / rich line APIs 的职责分裂

### Phase 3. 浏览器页面与演示
- 范围：`pages/*.ts`、`pages/*.html`、`pages/demos/**/*`
- 输出：页面模块卡、功能卡、demo 架构关系图
- 退出条件：能说明浏览器页面如何承担 accuracy、benchmark、corpus、probe 与 demo dogfooding

### Phase 4. 脚本与自动化校验
- 范围：`scripts/*.ts`、`shared/*.ts`
- 输出：校验/快照/诊断脚本卡；浏览器自动化链路说明
- 退出条件：能说明“页面 -> 自动化 -> JSON 快照/诊断输出”的闭环

### Phase 5. 数据、语料与研究素材
- 范围：`accuracy/*.json`、`benchmarks/*.json`、`corpora/*`、`thoughts.md`
- 输出：专题卡与风险卡，解释哪些数据是当前 source of truth，哪些只是历史背景
- 退出条件：能说明性能/准确性/语料 canary 的治理方式

### Phase 6. 设计评估与风险收束
- 范围：全仓库横向总结
- 输出：可借鉴设计、复杂点、脆弱点、未决问题与建议阅读路径
- 退出条件：总览、模块、功能、实现、专题、风险这六层互相闭环

## 当前策略
- 先完成 C0 与 C1，建立稳定导航与术语表
- 接着优先深挖 `src/`，因为它决定 Pretext 的核心产品论点
- `pages/` 与 `scripts/` 研究采用“页面功能 -> 调用库 API -> 自动化脚本消费页面”的顺序，避免重复

## 当前已知判断
- Pretext 的核心价值不是“排版渲染”，而是“浏览器字体度量驱动的、多语言、可缓存、多行文本布局预测”
- 架构中心是冷路径 `prepare()` 与热路径 `layout()` 的强分离
- rich line APIs 不是独立引擎，而是在同一 line-break 语义之上增加范围/文本物化能力
